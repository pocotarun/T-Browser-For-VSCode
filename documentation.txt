SECTION A: EXECUTIVE SYSTEM OVERVIEW & PHILOSOPHY
--------------------------------------------------------------------------------

A.1 THE ELECTRON SANDBOX LIMITATION
VS Code is fundamentally an Electron application, meaning it utilizes Chromium 
rendering processes wrapped in Node.js IPC channels. While extensions are granted 
access to the `Webview` API—essentially a sandboxed `<iframe>`—this container is 
subject to extreme security constraints. When a generic extension attempts to 
load an external URI (e.g., `https://google.com`) directly into the `src` attribute 
of a Webview iframe, it fails silently or crashes. 

A.2 THE CAUSE OF FAILURE: CSP & X-FRAME-OPTIONS
The internet relies on `X-Frame-Options: DENY` or `SAMEORIGIN` and strict 
`Content-Security-Policy` (CSP) headers to prevent Clickjacking. Because the 
VS Code Webview operates under a unique, local `vscode-webview://` protocol 
scheme, external servers instantly recognize it as a cross-origin threat and 
abort the connection at the networking level.

A.3 THE PROXY VIRTUALIZATION PARADIGM
To achieve absolute freedom and render any site seamlessly, T Browser does not 
attempt to fight the Chromium renderer. Instead, it deploys a completely isolated 
Node.js HTTP/HTTPS server in the background context of the extension. The UI 
iframe is instructed to load `http://127.0.0.1:PORT`. Because the target is 
technically a local network request, VS Code permits it. The background server 
then acts as an aggressive "Man-in-the-Middle" (MITM) proxy, fetching the requested 
external assets, stripping all restrictive security headers, spoofing the user-agent, 
and piping the sanitized binary/text data back to the iframe.

SECTION B: DETAILED BACKEND PROXY CORE
--------------------------------------------------------------------------------

The core virtualization engine is written in pure TypeScript (`src/extension.ts`) 
and operates on an asynchronous event-driven architecture.

B.1 THE REFERER MAPPING ENGINE (ELIMINATING THE 404 BUG)
A major architectural flaw in standard proxy implementations is the "Relative Path 
Collapse". When a Single Page Application (SPA) like React or Angular loads via 
the proxy, its JavaScript inevitably attempts to fetch relative internal APIs 
(e.g., `fetch('/api/user/profile')`). Because the SPA is running inside our 
`127.0.0.1` wrapper, the browser sends the request to `http://127.0.0.1:PORT/api/user/profile`. 
Without intervention, the local proxy returns a 404 Not Found.

The Referer Mapping Engine solves this definitively:
1. All explicit proxy requests are forcefully prefixed with a unique namespace: 
   `/__tbrowser_core_stream__/?url=`.
2. When the local Node server receives an incoming request that lacks this prefix, 
   it categorizes the request as an "Implicit Relative Asset".
3. The engine immediately halts execution and parses the `Referer` header of the 
   incoming request (which will point to the previous active URL).
4. It decodes the origin from the Referer, maps it to the implicit relative path, 
   and reconstructs the absolute target URL.
5. The request is dynamically re-routed downstream, entirely invisible to the user.

B.2 ENTERPRISE COOKIE JAR (STATEFUL SESSION MANAGEMENT)
Authentication requires state. If a proxy simply pipes data without managing cookies, 
sessions drop immediately on page transitions.
1. The `EnterpriseCookieJar` is initialized as a global nested Map: 
   `Map<Hostname, Map<CookieName, ICookieEntry>>`.
2. As the proxy receives upstream responses, it hooks into the `set-cookie` header array.
3. Every cookie string is boundary-split by the semicolon (`;`) delimiter. The primary 
   key-value pair is extracted securely, omitting strict flags like `Domain` or `Secure`.
4. When the user makes a subsequent outbound request to the same hostname, the 
   `serializeCookiesForRequest` method sweeps the jar, assembles the payload, and 
   forcefully injects it into the outgoing `Cookie` header.

B.3 DEEP STREAM ASSEMBLY (TAG CLIPPING PROTECTION)
Modern web servers utilize compression (gzip, deflate, brotli) and chunked 
transfer encoding. If the proxy attempts to run Regex string replacements on incoming 
network chunks as they arrive, it risks destroying HTML structure. For example, the tag 
`<iframe src="https://...">` might be split in half across two TCP packets: `<ifr` and 
`ame src="...">`. 
1. The `HTMLStreamAssemblyLayer` intercepts the response.
2. It detects the `Content-Encoding` and mounts the appropriate native `zlib` stream 
   decoder (e.g., `zlib.createBrotliDecompress()`).
3. It pushes all binary data chunks into a secure `Buffer[]` array.
4. Only upon receiving the `end` event does it execute `Buffer.concat()`, transforming 
   the entire payload into a single, cohesive UTF-8 string.
5. Absolute Regex mutations are then applied globally without fear of clipping.

SECTION C: ADVANCED CLIENT-SIDE OVERRIDING ENGINE
--------------------------------------------------------------------------------

While backend parsing is robust, it cannot intercept URLs dynamically constructed 
inside JavaScript execution contexts at runtime. Therefore, a massive client-side 
engine is generated and injected immediately after the `<head>` of every DOM payload.

C.1 THE INJECTED SCRIPT PAYLOAD (`JavascriptRuntimeInjector`)
This self-executing anonymous function operates directly within the browser context of 
the target site. It overrides native browser APIs to enslave their routing parameters.

C.2 OVERRIDING THE FETCH API & XMLHTTPREQUEST
1. The native `window.fetch` pointer is copied to `nativeWindowFetch`.
2. A proxy wrapper is defined that intercepts the URL parameter (whether it's a raw 
   string or a `Request` object).
3. The URL is passed through `resolveToAbsoluteProxyUrl()`, forcing it into the 
   `/__tbrowser_core_stream__` bridge.
4. The modified parameters are applied back to the native function. 
5. The exact same override pattern is applied to `XMLHttpRequest.prototype.open`.

C.3 OVERRIDING FORM SUBMISSIONS
1. `HTMLFormElement.prototype.submit` is hijacked to mutate the `action` attribute.
2. A passive, capture-phase `submit` EventListener is attached to the global `window` 
   to catch declarative form submissions triggered by UI buttons.

C.4 CONTINUOUS MUTATION OBSERVER
Dynamic SPAs constantly mount and unmount elements. 
1. A `MutationObserver` is initialized on `document.body` tracking `childList` and `subtree`.
2. As nodes are added (e.g., infinite scroll loading new `<img src="...">`), the 
   observer inspects tags like `SCRIPT`, `IFRAME`, `AUDIO`, `VIDEO`, and `A`.
3. It performs a real-time `src` or `href` rewrite, trapping the media inside the proxy.

SECTION D: UI/UX COMMUNICATION PROTOCOLS
--------------------------------------------------------------------------------

The frontend `ui.html` layout relies on strict DOM isolation. It communicates with the 
engine entirely through IPC (Inter-Process Communication).

D.1 POSTMESSAGE TELEMETRY DATA STREAMS
The injected runtime engine contains explicit telemetry dispatchers:
1. When the `load` event fires within the iframe, it executes:
   `window.parent.postMessage({ instruction: 'tbrowser-runtime-loaded' }, '*');`
2. The `EnterpriseFrontendController` in `ui.html` listens for this payload and 
   terminates the CSS3 Progressive Loading Bar simulation, ensuring the visual feedback 
   perfectly aligns with the actual network lifecycle.

D.2 HISTORY POINTER STATE MUTATION
To build a functional Back/Forward navigation stack without triggering full page 
reloads of the parent Extension environment:
1. An array (`navigationHistory`) and an index (`navigationPointer`) are maintained 
   in memory.
2. Every explicit routing execution pushes to the array and increments the pointer.
3. Every Back/Forward click decrements/increments the pointer and instructs the 
   iframe to load the cached string array value.
4. The active/disabled states of the SVG Toolbar buttons are reactively synced to 
   the boundaries of the pointer index.
